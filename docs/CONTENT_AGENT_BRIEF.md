# VisionTree Content OS 管理 Agent 说明

更新时间：2026-05-09

## 1. 本项目是什么

`content-system` 是 VisionTree 主项目的营销内容创作管理系统。它不是主产品本身，而是为了支持 VisionTree V2 早期内容启动而建立的内部工作台。

它的任务是把上游项目资料转化成可执行的内容生产系统：

- 将 VisionTree 的主叙事、口径边界和内容素材整理到素材池。
- 将不同账号适合使用的写法沉淀为模板库。
- 将第一周内容节奏拆成计划日历。
- 让创作者可以从素材和模板出发生成草稿、保存到内容池、上传媒体附件。
- 在发布后记录真实发布时间、平台指标和互动活动。

后续管理 agent 接手时，应把它理解为“营销内容运营中台”，而不是单纯的 CRUD demo。

## 2. 主项目认知

VisionTree 当前最稳妥的对外定义是：

- 认知增强引擎

更完整的理解是：VisionTree 是一个帮助人更会思考的认知增强项目。它回应的是 AI 时代基础脑力劳动被压缩之后，用户如何重新建立判断能力、结构化思考能力和复杂问题处理能力。

核心立场：

- 不是让 AI 替用户思考，而是让 AI 帮用户更会思考。
- 不是让 AI 记住用户喜欢什么，而是帮助用户明确“按什么框架思考”。
- 不是只给长答案，而是在关键节点停下来，让用户参与判断。
- 不是做更强的答案机器，而是做更好的思考扶手。

## 3. Vision 与 Tree 的当前表达边界

### Vision

`Vision` 是当前对外传播重点。它应被解释为：

- 内部一致的视角 / 认知框架

内容创作中可以讲：

- 帮用户先确定“按什么框架思考”。
- 帮用户在一致的视角里推进复杂问题。
- 减少通用模型混杂上下文带来的平均化输出。
- 在关键节点保留用户判断位置。

### Tree

`Tree` 是长期方向。它可被解释为：

- 将洞察沉淀为可连接、可演化、可复用的认知结构。

但当前不要把 Tree 写成已经成熟落地的产品能力。可以轻讲 Tree 是长期闭环的一部分，不能把它当作当前第一卖点。

## 4. 当前绝对不要误写

后续任何素材、模板、草稿、发布文案都必须避开这些误区：

- 不写成“AI 替你做判断”。
- 不写成“AI 自动给出最优结论”。
- 不写成成熟第二大脑、知识管理工具或认知树产品。
- 不写成模型库、prompt 模板库、skill 平台或开发者工程框架。
- 不把“可计算模型”“思维模型”“方法论”抬成 VisionTree 的总定义。
- 不虚构已上线能力、客户案例、量化结果或产品成熟度。
- 不把哲学愿景直接写成当前产品事实。

如果一个内容为了好看而牺牲准确性，应回退重写。

## 5. 当前内容创作进度

内容创作刚刚开始。

当前系统内已经准备好基础运营骨架：

- 7 个账号 / persona。
- 24 条素材记录。
- 7 个格式模板。
- 11 条第一周发布计划。
- 内容池目前为空。
- 活动记录目前为空。
- 平台指标目前还没有真实沉淀。

这意味着后续管理 agent 的重点是把已有资料转化为第一批内容，而不是假定系统已经有稳定运营数据。

第一阶段目标：

- 从素材池中筛选最适合第一周启动的素材。
- 按账号定位套用模板生成草稿。
- 将草稿保存到内容池，形成可追踪记录。
- 发布后通过平台回传或同步接口写入 `publishedAt` 和真实指标。
- 根据已发布内容反向更新素材和模板的引用次数。

## 6. 七个账号的内容分工

### `thinking-lab`

定位：视觉 + 互动实验室。

内容重点：

- 思维模型图解。
- 决策对比图。
- 概念卡。
- 投票、偏差识别、两分钟实验。

写作要求：

- 三秒看懂。
- 少解释，多动作。
- 不要一张图塞太多字。
- 第一周目标是建立“视觉优先 + 互动实验”的第一印象。

### `milo`

定位：工程师号 / 产品建设者。

内容重点：

- 真实技术选择。
- 产品取舍复盘。
- 面临什么选项、最后选了什么、为什么。

写作要求：

- 短句、克制。
- 像工程师对工程师说话。
- 不写品牌口号，不写粉丝式吹捧。

### `ai-doubt`

定位：反 AI 怀疑者 / 公开怀疑笔记。

内容重点：

- 反 AI 短帖。
- AI 效率神话拆解。
- 认知外包观察。
- AI 工具发布帖下的短评。

写作要求：

- 短、刺、冷静、有事实。
- 不写阴谋论，不做人身攻击。
- 怀疑必须有事实支撑，不能只是情绪宣泄。

### `nora`

定位：普通人试用号 / 非技术用户。

内容重点：

- AI 工具真实使用感受。
- 普通用户的困惑、不确定和小发现。
- “省了时间但不确定花得更好”的真实体验。

写作要求：

- 真实、日常、不装懂。
- 不做工具排行榜，不装专家。
- 可以承认没看懂，也可以保留迟疑。

### `eli`

定位：造概念的慢思考号。

内容重点：

- 月度概念词。
- 同一概念的不同场景切入。
- 热点背后的默认假设。
- 认知、判断、价值相关长帖。

写作要求：

- 慢、稳、有思想密度。
- 不追热点，不写轻飘金句。
- 第一周少量深帖比高频发布更重要。

### `thinking-tree`

定位：一棵会说话的树。

内容重点：

- 树的成长日记。
- 树的观察。
- 年轮、根系、枝叶等隐喻。
- 和其他账号互动。

写作要求：

- 平静，像活了很久的树。
- 淡淡幽默。
- 不卖萌，不装可爱，不写成吉祥物。
- 这是品牌隐喻账号，不是成熟 Tree 产品证明。

### `visiontree`

定位：官方号 / 叙事收拢。

内容重点：

- 品类定义帖。
- 对比式表达。
- 转发矩阵账号并补一句评论。
- 官方叙事汇总。

写作要求：

- 清楚、克制、有定义感。
- 不抢其他账号的戏。
- 第一周重点是把六个账号收拢到同一条 VisionTree 叙事里。

## 7. 素材池怎么管理

素材池不是“收藏夹”。它是内容生产的事实与观点来源。

每条素材应尽量包含：

- `title`：素材标题。
- `theme`：服务哪个主题。
- `source`：来源，如 VT 素材库、会议纪要、项目文档、网页、PDF。
- `format`：网页、PDF、图片、笔记、访谈摘录等。
- `tags`：账号、主题、内容类型、传播入口。
- `summary`：可直接帮助创作的摘要。
- `notes`：创作提示和边界。
- `resources`：原始网页/PDF/本地文件链接。

管理要求：

- 素材必须能追溯来源。
- 摘要要面向创作，不只做资料复述。
- notes 要写清楚“适合怎么用”和“不要怎么误用”。
- 素材可以服务多个账号，但必须标明最适合的账号或场景。
- `referenceCount` 由已发布内容自动聚合，不手动写死。

## 8. 模板库怎么管理

模板库不是“万能提示词库”。它是不同账号、不同内容类型的稳定表达结构。

每个模板应尽量包含：

- `title`：模板名称。
- `format`：短帖、长帖、图解、投票、日记、官方定义等。
- `channels`：当前主要是 X。
- `hook`：开场方式。
- `structure`：内容结构步骤。
- `length`：建议长度。
- `notes`：适用账号、适用素材、禁忌写法。
- `resources`：可参考的说明资料。

管理要求：

- 模板要服务内容风格，不只是控制字数。
- 模板要能体现账号差异。
- 同一个主题给不同账号使用时，应选择不同模板，不要把七个账号写成同一种声音。
- `structure` 会被前端用于草稿生成，请保持简洁、可执行。
- `referenceCount` 由已发布内容自动聚合，不手动写死。

## 9. 内容池和发布数据规则

内容进入内容池时，后端写入 `savedAt`。这代表内容被保存为可管理记录的时间。

真实发布时间必须使用 `publishedAt`。它不能由普通草稿保存接口写入，应由发布同步或平台回传接口写入。

平台指标包括：

- 曝光 / views
- 点赞 / likes
- 评论 / comments
- 转发 / shares
- 收藏 / saves

这些指标不应由前端人工写死。当前演示发布接口会模拟数据，真实接入后应改为平台查询或回传。

## 10. 后续管理 agent 优先级

优先级从高到低：

1. 保持主项目叙事准确，不牺牲边界。
2. 把第一周计划转化为真实草稿和内容池记录。
3. 让每条内容都能追溯到素材和模板。
4. 发布后回填真实 `publishedAt` 和平台指标。
5. 根据发布反馈调整素材优先级和模板使用频率。
6. 再考虑扩展素材库、模板库和平台同步能力。
